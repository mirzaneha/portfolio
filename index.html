Artifact 01 — Workshop 1 — Assignment 1.4

From Questions to Insights — Building a Data Insights Assistant

AI Lab

A hands-on AI lab documenting the end-to-end experience of exploring LLMs, Custom GPTs, and research AI tools — culminating in the design and build of the Data Insights Assistant built on Chatbase.co. This artifact answers: can this person use AI tools to solve a real problem that data professionals face every day?

Objective

To gain direct hands-on experience with multiple AI tools and apply a Design Thinking framework to build a practical AI assistant that helps data analysts identify the right questions to ask their data before jumping into analysis.

Description

The Data Insights Assistant guides users through defining their business problem, identifying the right analytical approach, and asking better questions of their data — the step most analysts skip before opening their BI tool.

Process

1

LLM (Google Gemini): Ran same prompt across 3 separate conversations — each surfaced a different valuable dimension. Key lesson: run multiple chats and combine the best elements.

2

Custom GPT (Data Analyst GPT): Asked clarifying business context questions before making suggestions — producing targeted output I could use in a real project immediately.

3

STORM AI: Researched AI in Business Intelligence — multi-perspective approach produced a balanced article on natural language querying and the evolving analyst role.

4

Design Thinking → Build → Iterate: Questions-first approach over SQL generator. Three iterations refined the bot from generic to genuinely consulting-quality conversations.

Artifact 02 — Workshop 2 — Assignment 2.3

ML vs. Deep Learning — A Data Analyst's Comparative Study

Written Analysis

A comparative analysis of machine learning and deep learning through two real-world case studies — Microsoft Outlook's spam filter (Decision Tree/Random Forest) and Google's Diabetic Retinopathy Detection System (CNN). Written from a data analyst's perspective that asks: which approach produces results I can trust, explain, and act on?

Objective

To demonstrate the ability to differentiate between ML and DL by grounding the comparison in real applications that data professionals encounter — and to articulate why each approach is appropriate from the perspective of a practitioner who needs to explain and defend analytical choices to business stakeholders.

Description

The analysis examines Outlook's use of Decision Tree-based classification for spam filtering — where known features and interpretability requirements make traditional ML the right choice — and Google's CNN-based diabetic retinopathy detection — where raw medical image data and the need for automatic pattern discovery make deep learning the only viable approach.

Case Study 1 — ML: Outlook Spam Filter (Decision Tree / Random Forest)

✓ Why ML is Suitable

Binary classification with well-understood, structured features — sender reputation, keyword patterns, link density, attachment types. Decision trees produce interpretable rules Microsoft's security team can audit and explain to enterprise customers. Efficient at scale without DL computational overhead (Starmer, 2021).

✗ Why Deep Learning is NOT Suitable

Spam detection does not require automatic feature discovery — the features are already known and structured. A deep neural network would consume significantly more resources, produce less interpretable results, and likely not outperform a well-tuned Random Forest on this specific structured classification task.

Case Study 2 — Deep Learning: Google Diabetic Retinopathy Detection (CNN)

✓ Why DL is Suitable

Medical image analysis requires identifying subtle visual patterns — microaneurysms, hemorrhages, abnormal blood vessel growth — across high-resolution images with enormous variability. CNNs automatically learn hierarchical visual representations from raw pixels without manual feature engineering. Google's system matches specialist ophthalmologist accuracy (Couto, 2025).

✗ Why Traditional ML is NOT Suitable

Manual extraction of visual features is impractical for medical imaging. A decision tree fed hand-crafted image features would miss the subtle, distributed visual patterns that define early-stage diabetic retinopathy — resulting in dangerous false negatives in a clinical setting where accuracy is literally a matter of vision.

Concepts Demonstrated

Decision Trees Random Forest Convolutional Neural Networks Medical Image Analysis Feature Engineering Model Interpretability

Value Proposition

For analytics managers and business leaders, this artifact demonstrates the kind of structured analytical thinking that makes data professionals valuable in AI/ML decision-making. The comparison is not about which model is more technically impressive — it is about which model is trustworthy, explainable, and appropriate for the stakes involved. A data analyst who can make and defend that distinction is far more valuable than one who always reaches for the most sophisticated option.

Unique Value

My data analytics background shaped this analysis around a question I ask every day: can I explain this result to a stakeholder who did not build the model? For Outlook's spam filter, yes — the rules are transparent and auditable. For Google's retinopathy detector, the answer is more nuanced — and recognizing that nuance, and what it means for how the model should be deployed and governed, is exactly what data analysts bring to AI/ML decision-making.

Relevance

Data analysts are increasingly being asked to evaluate, validate, and communicate AI model outputs — not just build dashboards. Understanding whether a model uses interpretable ML or a black-box deep neural network changes how outputs should be validated, how errors should be investigated, and how results should be presented to decision-makers. This artifact demonstrates that understanding applied to real-world systems I would encounter in my career.
